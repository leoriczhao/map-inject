wenhao_plugin 逆向分析报告

基于 wenhao_plugin.tga（4.65MB x86 PE DLL）的二进制逆向。文件来源：内置 JAPI 1.34.w3x。

一、概述

结论先行：wenhao_plugin 不是一个自包含的 Lua 引擎，它是一个 MPQ 文件导出器 + DLL 加载器。
它将 3-5 个从 MPQ 提取的 DLL 链式加载，把 JASS 原生注册（jass.dll）和 Lua 引擎启动（lua_engine.dll）委托给这些子 DLL。

wenhao_plugin 的物理构成

PE 属性x86 DLL, Image Base 0x10000000
导出"call" (RVA 0x1100), "get" (RVA 0x1230)
导入KERNEL32.dll: FindResourceA, FreeResource, LoadResource, SizeofResource, LockResource, GetLastError, Sleep, GetProcAddress, LoadLibraryA, GetModuleHandleA
依赖msvcrt.dll, Storm.dll（运行时动态加载）
体积4,653,568 字节 — 其中大量是嵌入的 C++ RTTI、bzip2 压缩器代码、Lua 5.3 运行时

三、IAT 解析

DLL 从 KERNEL32.dll 导入以下函数，拆分为两组用法：

文件资源操作组

用于从自身 PE 资源中提取嵌入文件：

FindResourceA查找嵌入资源
SizeofResource获取资源大小
LoadResource加载资源
LockResource锁定资源内存

DLL 加载与流程控制组

值得注意的是 没有导入 VirtualAlloc、WriteProcessMemory 等内存操作函数。wenhao_plugin 本身不做进程注入，它的 job 纯粹是文件导出 + DLL 加载。

六、sub_10001020 — MPQ 文件导出重试包装器

这是 "call" 调用的核心辅助函数，封装了 MPQ 文件导出 + 重试逻辑：

// 从反汇编还原的伪代码：
// 参数: edi = 文件名, edx = 扩展名, ecx = 操作码
bool sub_10001020(const char* filename, const char* extension, char opcode) {
    uint32_t pExport = *0x1000E028;  // 全局 pExportFromMpq 指针

    for (int retry = 0; retry < 10; retry++) {
        pExport(filename);            // 调用 game.dll!ExportFileFromMpq(filename)
        uint32_t err = GetLastError();

        if (err == 0x7E)  goto success;  // ERROR_MOD_NOT_FOUND (文件已存在？)
        if (err == 0x00)  goto success;  // ERROR_SUCCESS
        if (err == 0x02)  goto success;  // ERROR_FILE_NOT_FOUND?

        // 失败但可重试：
        if (err != 0x7E && err != 0x02 && err != 0x00) {
            Sleep(10);                   // 等待 10ms
            // 使用 opcode + extension 做额外文件操作...
        }
    }

success:
    // 对导出的文件进行后续处理
    // 使用 LoadResource / LockResource / FreeResource 管理资源
    return (err == 0x7E || err == 0x02 || err == 0x00);
}

关键发现：ExportFileFromMpq 的函数指针来自全局变量 0x1000E028，不是 "call" 的参数。这意味着在 DllMain 或某个初始化器中，需要先把 pExportFromMpq 写入这个全局槽位。但 "call" 的参数 pExportFromMpq 被保存到了 0x10014F54（另一个全局变量），可能用于不同的代码路径。

九、与 YDWE 的血缘关系

wenhao_plugin 内部代码结构

从 C++ RTTI 符号还原：

  ├── func_value        (函数元数据)
  ├── global_variable   (全局变量访问)
  ├── string_fake       (字符串包装)
  └── hashtable::       (哈希表)

  ├── replace_memory    (内存补丁)
  └── window_event      (窗口事件)

base::lua_engine::slk::
  ├── slk_interface_storm
  ├── ObjectManager
  ├── SlkValue / SlkSingle

YDWE 当前代码结构（ydwar3 + yd_lua_engine）

  ├── call_param        ✓ 相同
  ├── func_value        ✓ 相同

核心发现：wenhao_plugin 和 YDWE 共享同一个 C++ 代码谱系。wenhao_plugin 将 JASS 注册（jass.dll）和 Lua 引擎（lua_engine.dll）分离为两个独立 DLL，由 wenhao_plugin 作为加载器统一管理。

YDWE 后来将这两个功能合并到 yd_lua_engine.dll 中，并添加了 virtual_mpq 的 force_watch 基础设施。但 jass.dll 和 lua_engine.dll 的内部逻辑（jass.* 模块、SLK 系统、JASS 原生注册）在 YDWE 中得以保留和发展。

十、你的方案为什么跑不了

现在有了 wenhao_plugin 的完整逆向，对比你的 port 方案就很清楚了：

核心问题

wenhao_plugin 的 lua_engine.dll 的 "init_lua_engine" 导出一被调用就立即创建 Lua 状态并加载执行 war3map.lua。它是一个主动的、立即生效的函数。

YDWE 的 yd_lua_engine.dll 的 Initialize() 导出只注册了一个等待型回调。它依赖 virtual_mpq IAT hook 在将来某个时刻触发，而那个时刻在 callback 上下文中永远不会到来。

这两个 DLL 虽然代码同源，但初始化接口的设计哲学完全不同。

修法

最简单的修复：在 YDWE 的 lua_loader.cpp 中添加一个新导出，等效于 wenhao_plugin 的 "init_lua_engine"：

// 新增：直接触发 Lua 加载，不依赖 force_watch
// 等效于原版 lua_engine.dll 的 "init_lua_engine" 导出
extern "C" __declspec(dllexport) void InitLuaEngine() {
    initialize_lua();  // 直接从 MPQ 读 war3map.lua + 执行
}

// Initialize 保持不变（保留 force_watch 注册，用于 YDWE 启动器场景）
extern "C" __declspec(dllexport) void Initialize() {
    lua_loader::initialize();  // 注册 force_watch + event_game_reset
}

然后在 callback 中按 wenhao_plugin 原版的顺序调用：

// 1. 加载 DLL 链（你已实现）
// 2. 初始化 JAPI native（你已实现）
// set i = GetModuleProcAddress("yd_jass_api.dll", "Initialize")
// call CallCdeclWith1Args(i, 0)

// 3. 初始化 Lua 引擎注册（保留 force_watch）
// set i = GetModuleProcAddress("yd_lua_engine.dll", "Initialize")
// call CallCdeclWith1Args(i, 0)

// 4. ★ NEW：直接触发 Lua 加载（等效原版 init_lua_engine）
