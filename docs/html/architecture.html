🏗️ 架构总览

1. 项目目标

用一个 japi.dll（静态链接 Lua 5.3）替代 YDWE 的五 DLL 链（lua53.dll → ydbase.dll → ydwar3.dll → yd_lua_engine.dll → yd_jass_api.dll），通过 callback exploit 加载，在游戏内自启动 Lua 引擎，提供完整的 JASS↔Lua 互操作。

2. 核心原理：Callback Exploit

Warcraft III 的 StartCampaignAI 函数接受一个 JASS 文件名作为参数，由 AI 编译器编译执行。关键漏洞在于 jasshelper 的类型混淆：利用 l__ 前缀绕过类型检查，使得 JASS 代码可以读写任意内存地址。这就是所谓的 "MemHackAPI"。

核心洞察： callback 文件虽然以 JASS 语法编写，但通过 l__ 前缀的数组类型混淆，获得了 C 语言级别的任意内存读写能力。这使得纯 JASS 代码可以调用 LoadLibrary、GetProcAddress 等 Windows API。

3. 完整执行流程

1

War3 加载地图 → 编译 war3map.j → 执行 main()

地图脚本 war3map.j 的 main() 函数调用 StartCampaignAI(Player(12), "callback")，触发 AI 编译器加载并编译 callback 文件。AI 编译器跳过常规类型检查，允许危险的内存操作。

2

callback::main() — 版本检测 + 内存解锁 + DLL 加载

callback 的 main() 执行：(a) 版本检测（比较 PE 头识别 1.27a/1.26a/1.24b/1.24e）；(b) UnlockMemory 获取任意内存读写；(c) AllocateExecutableMemory 分配 RWX 内存；(d) ExportFileFromMpq 从 MPQ 导出 japi.tga；(e) LoadLibrary 加载 DLL；(f) GetProcAddress("call") 获取导出函数；(g) CallCdecl(call, pExportFromMpq) 进入 C++ 世界。

3

japi.dll::call(pExportFromMpq) — JASS VM 上下文建立

dllmain.cpp:16：计算 pJassEnvAddress = game.dll + 0xBE3740，调用 set_vm_context_from_env() 执行三级指针链 [[[pJassEnvAddress]+20]+144]+4 获取 VM 上下文指针。

4

init_native_registry() — 扫描 + 注册自定义 native

native_registry.cpp:158：扫描 game.dll .text 段，从 "Deg2Rad" 字符串引用定位第一个 NativeRegBlock，顺序遍历获取全部 1167 个 JASS natives。然后通过 nfunction_add 注册 3 个测试 native（TestAdd, DebugMsg, SayHello）。

5

init_lua_bootstrap() — Lua 引擎引导 + war3map.lua

lua_bootstrap.cpp:281：创建 lua_State、注册全部 6 个 jass.* 预加载模块、从 MPQ 读取并执行 war3map.lua（用户的游戏逻辑脚本）。

5. 关键设计决策

5.1 为何是单 DLL？

YDWE 使用五 DLL 链：lua53.dll → ydbase.dll → ydwar3.dll → yd_lua_engine.dll → yd_jass_api.dll。每个 DLL 有独立的导出表、重定位表和 CRT 依赖。japi.dll 将 Lua 5.3 静态链接进 DLL，配合 /MT（静态 CRT），产出单个 484KB 的 DLL，零外部依赖（仅需 kernel32.dll 和 Storm.dll）。

5.2 为何扫描 .text 段而非 Hijack？

YDWE 通过 Hijack storm_alloc + TlsGetValue 间接获取 nfunction_add 地址。japi.dll 直接在 game.dll 的 .text 段扫描 20 字节的 NativeRegBlock 模式（push/mov edx/mov ecx/call 序列），从 "Deg2Rad" 字符串引用反向定位第一个注册块后顺序遍历。这种方法：(a) 不依赖特定 WC3 版本的硬编码偏移；(b) 自动发现全部 1167 个 natives（vs. YDWE 的硬编码数十个）；(c) 代码量更少。

5.3 Lua metatable __index vs. Hashtable + UnitId Hook

内置 JAPI 使用 hashtable + UnitId hook 实现 native 调度——每次调用需查 hashtable 和 JASS 的 UnitId 网关。japi.dll 使用 Lua 的 metatable __index：所有 1167 个 native 通过 jass.NativeName(...) 直接调用，__index 自动查找并返回 wrapper closure。性能更好（Lua 内部 hash vs. JASS hashtable），语法更自然。

5.4 fake jstring vs. 原生 create_string

JASS VM 内部 jstring 有特定内存布局。japi.dll 实现了手动构造的 string_fake RAII 类（jass_types.h:42）：8 个 uint32_t 的栈上数组，memory_[2] 自引用，memory_[7] 指向 C 字符串。同时通过 "I2S" native 的引用链扫描找到了 game.dll 的真 create_string 函数（__fastcall），用于需要持久生命周期的字符串。
