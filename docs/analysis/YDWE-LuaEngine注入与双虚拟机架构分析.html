YDWE LuaEngine 注入与双虚拟机架构分析

从进程注入到双虚拟机并行运行 —— 完整技术链路拆解。分析基于源码阅读，涵盖 21 个核心源文件。

一、总体概述

YDWE 在不修改 Game.dll 二进制文件的前提下，通过代码洞注入将一个完整的 Lua 5.4 运行时注入到 war3.exe 进程中，与 Warcraft III 原生的 JASS 虚拟机并行运行，形成"双虚拟机"架构。

四个阶段的流水线

Phase 1 进程注入 — war3.exe 挂起创建，shellcode 注入 LuaEngine.dll

Phase 2 Lua 引导 — war3/main.lua 通过 IAT Hook 拦截 Game.dll 加载

Phase 3 桥接建立 — yd_lua_engine.dll 构建 JASS↔Lua 双向通道

Phase 4 双 VM 运行 — JASS 与 Lua 通过 trampoline/coroutine 并发

核心不变量

✅ 不修改 Game.dll 任何字节

✅ 不依赖 CreateRemoteThread（规避反外挂）

✅ 支持 1.27a/b 等多版本（运行时模式匹配）

✅ 纯 Lua 地图 + JASS/Lua 混合地图双模式

2.2 代码洞注入机制

核心思路：不用 CreateRemoteThread（会被反外挂检测），而是直接操作目标进程主线程的 EIP/RIP 寄存器，让 shellcode 在主线程上下文中执行。
x86 注入流程（injectdll_x86） 
1. SuspendThread(hThread)
2. VirtualAllocEx → 为 shellcode 分配可执行内存
3. VirtualAllocEx → 为 DLL 路径字符串分配内存
4. 将 LoadLibraryW 地址 + DLL路径指针 + 原始EIP 写入 shellcode
5. SetThreadContext → EIP = shellcode 地址
6. ResumeThread(hThread)

Shellcode 执行流程（39 字节）：

push EIP_original        ; 保存返回地址
pushfd                   ; 保存标志寄存器
pushad                   ; 保存通用寄存器
push [ptrDllPath]        ; DLL 路径指针入栈
call [LoadLibraryW]      ; 加载 LuaEngine.dll
popad                    ; 恢复通用寄存器
popfd                    ; 恢复标志寄存器
ret                      ; 跳回原始 EIP，继续 war3.exe 正常启动
x64 注入流程（injectdll_x64）
  • 通过 wow64ext.h 从 32 位进程操作 64 位进程
  • 使用 LdrLoadDll（从 ntdll.dll）替代 LoadLibraryW
  • Shellcode 保存/恢复完整 x64 寄存器上下文（push/pop r* 系列）

DLL 选择逻辑：

bool injectdll(PROCESS_INFORMATION& pi, wstring x86dll, wstring x64dll) {
    if (is_process64(pi.hProcess))
        return injectdll_x64(pi, x64dll);  // 64位war3 → 64位LuaEngine
    else
        return injectdll_x86(pi, x86dll);  // 32位war3 → 32位LuaEngine
}

三、第二阶段：LuaEngine.dll 启动与 Lua 引导系统

3.1 LuaEngine.dll 入口

当 shellcode 中 LoadLibrary("LuaEngine.dll") 执行时，DllMain 被调用：

// DllMain.cpp
BOOL APIENTRY DllMain(HMODULE, DWORD reason, LPVOID) {
    if (reason == DLL_PROCESS_ATTACH) {
        std::wstring name = getenv(L"ydwe-process-name"); // "war3"
        lua_State* L = LuaEngineCreate(name.c_str());
        if (L) {
            LuaEngineStart(L);  // require "main"
        }
    }
}

LuaEngineCreate 关键步骤

  1. Hook lua_pcallk → FakeLuaPcall：用 __try/__except 包裹所有 Lua 调用，防止崩溃传播到 war3 进程
  2. luaL_newstate() 创建 Lua 5.4 状态机
  3. 打开所有标准库（table, string, math, coroutine 等）
  4. 设置 package.path → script/common/?.lua;script/war3/?.lua
  5. 设置 package.cpath → ydwe/bin/?.dll
void LuaEngineStart(lua_State* L) {
    lua_pushcfunction(L, error_handler);
    lua_getglobal(L, "require");
    lua_pushstring(L, "main");
    lua_pcall(L, 1, 0, -3);  // require "main" → 加载 script/war3/main.lua
}

3.2 Lua 引导脚本：war3/main.lua

script/war3/main.lua 是整个插件系统的编排器，三个步骤：

Step A — IAT Hook LoadLibraryA

拦截 kernel32!LoadLibraryA，当检测到正在加载 Game.dll 时：

  • 可选加载补丁版 Game.dll
  • 打开 patch.mpq
  • 发布事件 'GameDll加载'（携带 Game.dll 模块句柄）

Step B — 扫描加载插件 DLL

响应 'GameDll加载' 事件，三个并行处理器：

  1. Hook Game.dll 的 CreateWindowExA → 检测魔兽窗口创建
  2. 加载 yd_loader.dll（IAT Hook 加载器），挂载战争补丁
  3. 扫描 plugin/warcraft3/ 目录，逐个加载所有 .dll，并调用其 Initialize() 导出函数

Step C — 事件驱动初始化

后续各插件（yd_lua_engine、yd_jass_api 等）在各自的 Initialize() 中向引导系统注册回调，挂载到 'GameDll加载'、'窗口初始化'、'游戏重置' 等事件上，形成完整的事件驱动架构。每个插件只关心自己的初始化时机，无需感知其他插件的存在。

四、第三阶段：JASS-Lua 桥接（yd_lua_engine.dll）

4.1 yd_lua_engine.dll 初始化

由 war3/main.lua 通过 LoadLibraryW + GetProcAddress("Initialize") 加载。核心初始化逻辑通过 虚拟 MPQ 监视器 检测地图加载：

void lua_loader::initialize() {
    // 监视 war3map.j 的 MPQ 访问 —— 这是地图 JASS 脚本
    virtual_mpq::force_watch("war3map.j", [&](...) {
        if (地图未变化) return false;
        initialize_lua();  // 初始化地图 Lua 环境
        return false;
    });

    // 游戏重置时销毁 Lua 状态
    event_game_reset([&]() {
        lua_close(mainL);
        mainL = nullptr;
    });
}

4.2 地图类型检测 —— 双模式分支

void initialize_lua() {
    if (地图MPQ中存在 "script\\war3map.lua") {
        // ═══ 纯 Lua 模式 ═══
        lua_State* L = getMainL();        // 创建地图 Lua 状态
        luaL_openlibs(L);                 // 打开标准库
        open_lua_engine(L);               // 注册 jass.* 模块
        runtime::initialize();            // 运行时配置
        luaL_loadbuffer(L, war3map.lua);  // 加载地图 Lua 脚本
        safe_call(L, 0, 0, true);         // 执行
    } else {
        // ═══ JASS + Lua 混合模式 ═══
        jass::table_hook("Cheat", &RealCheat, FakeCheat);
        jass::japi_table_add(EXExecuteScript, "EXExecuteScript", "(S)S");
    }
}
地图包含 war3map.lua → 纯 Lua 模式，Lua 是主脚本语言；地图只有 war3map.j → 混合模式，通过 Cheat Hook 和 EXExecuteScript 在 JASS 和 Lua 之间架桥。

5.1 两个独立的 Lua 状态

状态机创建者用途

A) Lua 调用 JASS 原生函数

当 Lua 代码写 local t = CJ.CreateTrigger() 时：

1. jass.common 的 __index 元方法被触发 → jass_get(L)
2. 在 JASS 函数表中查找 "CreateTrigger"
3. 返回 C closure (upvalue = jass::func_value* 指针)
4. 调用时进入 jass_call_closure(L)
     └─ jass_call_native_function(L, nf, func_address)
          ├─ 从 Lua 栈读取参数，通过 jassbind 转换为 JASS 类型
          │    boolean → TYPE_BOOLEAN
          │    number  → TYPE_INTEGER / TYPE_REAL
          │    string  → TYPE_STRING (通过 jass::create_string)
          │    userdata/lightuserdata → TYPE_HANDLE
          │    function → TYPE_CODE (通过 trampoline 创建)
          ├─ 构造 jass::call_param 参数列表
          ├─ [可选] SEH 包裹：__try { jass::call(...) } __except { ... }
          └─ 返回值通过 jass_push 转换回 Lua

B) Sleep 感知的 Lua→JASS 调用

TriggerSleepAction、TriggerWaitForSound 等函数会挂起 JASS 执行。特殊处理：

int jass_call_closure(lua_State* L) {
    int result = jass_call_native_function(L, ...);
    if (lua_isyieldable(L)) {
        jass_vm_t* thread = get_jass_thread();
        if (thread && thread->has_sleep) {
            // 回退 JASS 指令指针（跳过 trampoline 占用的 9 条 opcode）
            thread->opcode -= jass::trampoline_size() / sizeof(jass::opcode);
            return lua_yield(L, 0);  // Lua coroutine 让出执行权
        }
    }
    return result;
}

C) JASS 调用 Lua 函数（Trampoline 机制）

当 Lua 将函数作为 code 参数传给 JASS 时（如 TriggerAddAction），需要 JASS 可调用的入口。

Trampoline 创建：

cfunction_to_code(L, index)
  ├─ runtime::callback_push(L, index) → Lua 函数注册到回调表，获得 ref
  ├─ jass::trampoline_create(jass_callback, lua_State*, ref)
  │    ├─ 分配可执行内存
  │    ├─ 生成 9 条 JASS opcode（跳板字节码）
  │    └─ 返回 opcode 地址 → 这就是 JASS 中的 code 值
  └─ 返回 code handle

汇编存根（lua_to_nativefunction.h）：

push esi              ; 保存 esi
mov  esi, esp
add  esi, 8           ; esi → 参数列表指针
push esi              ; 压栈参数列表地址
mov  ecx, this        ; 设置 C++ this 指针
call lua_function     ; 调用 C++ → C++ 再调 Lua
pop  esi              ; 恢复 esi
ret

这个存根符合 JASS 调用约定（__stdcall-like，参数指针在 ESI），可以直接放在 JASS 原生函数表中。

D) JASS 调用 Lua 表达式（EXExecuteScript） 
jstring_t EXExecuteScript(jstring_t script) {
    lua_State* L = getMainL();
    std::string str = fmt::format("return ({})", from_string(script));
    luaL_loadbuffer(L, str.c_str(), str.size(), ...);
    safe_call(L, 0, 1, true);
    jstring_t result = 0;
    if (lua_isstring(L, -1))
        result = jass::create_string(lua_tostring(L, -1));
    lua_pop(L, 1);
    return result;
}

JASS 侧用法：

local string result = EXExecuteScript("my_lua_function(42)")

5.4 并发执行模型

双虚拟机的"并行"实际上是协作式并发（cooperative concurrency），而非真正的多线程并行。控制权通过 trampoline 进入 Lua，通过 yield 归还 JASS。
