2. Native 扫描：NativeRegBlock 20 字节模式

每个 JASS native 在 game.dll 的 .text 段中有对应的 20 字节注册代码块：

NativeRegBlock 内存布局（jass_vm.cpp:116）

#pragma pack(push, 1)
struct NativeRegBlock {
    uint8_t  push_op;       // 0x68          — PUSH opcode
    uint32_t param_ptr;     // → .rdata     — 参数字符串指针 "(II)I"
    uint8_t  mov_edx;       // 0xBA          — MOV EDX opcode
    uint32_t name_ptr;      // → .rdata     — native 名称字符串指针 "Deg2Rad"
    uint8_t  mov_ecx;       // 0xB9          — MOV ECX opcode
    uint32_t func_addr;     // → .text      — native 函数地址
    uint8_t  call_op;       // 0xE8          — CALL opcode
    uint32_t call_rel;      // rel32         — 相对偏移到 nfunction_add

    bool verify() const {
        return push_op == 0x68 && mov_edx == 0xBA
            && mov_ecx == 0xB9 && call_op == 0xE8;
    }
};
#pragma pack(pop)

等价的 x86 汇编

68 XX XX XX XX    push  param_str_ptr     ; 参数规格字符串
BA XX XX XX XX    mov   edx, name_ptr      ; native 名称
B9 XX XX XX XX    mov   ecx, func_addr     ; native 函数地址
E8 XX XX XX XX    call  nfunction_add     ; 注册到 JASS VM

扫描过程（scan_natives, L139）

  1. s.search_string("Deg2Rad") — 在 .rdata 找 "Deg2Rad"，再在 .text 找引用它的 DWORD
  2. block = (NativeRegBlock*)(ref - 6) — 向前偏移 6 字节定位到 NativeRegBlock 起始（DWORD 引用的位置在 mov edx, name_ptr 指令的第 2-5 字节）
  3. block->verify() — 验证 20 字节魔数
  4. while (block->verify()) { ... block++; } — 顺序遍历所有连续的注册块，提取 name/address/param
  5. 保护：if (count > 10000) break

扫描结果

当前扫描到 1167 个 JASS natives，存储在 g_native_map（name → NativeInfo）和 g_native_list（顺序列表）中。通过 find_native(name)、get_native_info(name)、for_each_native(cb) 访问。

四级指针链： [[[game.dll + 0xBE3740] + 0x14] + 0x90] + 0x04

等价于 callback 中的 GetJassContext(1)。所有 JASS native 调用都需要 VM context 作为 ecx（thiscall 风格的 this 指针）。SEH 异常处理确保指针链中任一级为 NULL 时不会崩溃。

4. Native 调用 Trampoline

jass::call（jass_call.cpp:10）

uintptr_t call(uintptr_t func_address, const uintptr_t* param_list, size_t count) {
    uintptr_t retval = 0;
    size_t param_size = count * sizeof(uintptr_t);
    uintptr_t vm_ctx = vm::get_vm_context();
    uintptr_t saved_esp = 0;

    __asm {
        mov  eax, func_address    ; 目标函数
        mov  edx, vm_ctx          ; JASS VM context → ecx (thiscall this)
        mov  saved_esp, esp       ; ★ 保存原始 ESP

        sub  esp, param_size      ; 分配参数空间
        mov  edi, esp             ; 目标 = 栈顶
        mov  esi, param_list      ; 源 = 参数数组
        mov  ecx, count           ; 计数
        cld
        rep  movsd                ; 复制参数到栈

        mov  ecx, edx             ; VM context = this 指针
        call eax                  ; 调用 native
        mov  retval, eax          ; 保存返回值

        mov  esp, saved_esp       ; ★ 恢复 ESP（兼容 cdecl/stdcall/thiscall）
    }
    return retval;
}
关键设计：ESP 保存/恢复。 JASS natives 使用混合调用约定（__cdecl/__stdcall/__thiscall），callee 的栈清理量不确定。通过保存调用前的 ESP 并在返回后恢复，无论 native 使用哪种约定都能正确返回。参数通过 sub esp 分配 + rep movsd 复制。

5. fake jstring 内存布局

string_fake（jass_types.h:42）+ from_string（jass_vm.cpp:56）

class string_fake {
    uint32_t memory_[8];  // 32 字节栈上数组
public:
    string_fake(const char* str) {
        memory_[2] = (uint32_t)&memory_[0];  // [outer+8] = &outer (自引用)
        memory_[7] = (uint32_t)&str[0];      // [outer+28] = C 字符串指针
    }
    operator jstring_t() const { return (jstring_t)memory_; }
};

// 逆向读取：还原 C 字符串
const char* from_string(jstring_t val) {
    val = *(uint32_t*)((uint32_t)val + 8);   // 解引用自引用
    return (const char*)(*(uint32_t*)(val + 28));  // 读字符串指针
}

create_string（真字符串分配器）

通过 "I2S" native 的引用链在 game.dll 中定位真正的字符串分配器（jass_vm.cpp:258-330）：

  1. search_string("I2S") → 找到 .text 中引用 "I2S" 的位置
  2. 跳过 push 指令（5 字节）
  3. 找第一个 CALL (E8) → 第一个辅助函数
  4. 跳过第一个 CALL（5 字节）
  5. 找第二个 CALL (E8) → create_string
  6. 解析相对地址：target = addr + 5 + rel32

签名：__fastcall jstring_t create_string(const char* str)，用于需要持久生命周期的字符串。

6. x86 Thunk 生成

ThunkCode（lua_japi.cpp:24，lua_hook.cpp:32 重复）

#pragma pack(push, 1)
struct ThunkCode {                    // 共 14 字节
    uint8_t  push_op;      // 0x68
    uint32_t native_id;    // 唯一标识符（或 hook_id）
    uint8_t  call_op;      // 0xE8
    uint32_t call_rel;     // 相对偏移 = dispatcher - (thunk + 10)
    uint8_t  add_esp[3];   // 0x83 0xC4 0x04
    uint8_t  ret_op;       // 0xC3
};
#pragma pack(pop)
static_assert(sizeof(ThunkCode) == 14, "ThunkCode must be 14 bytes");

Thunk 作用： 为每个注册的 JAPI native 提供唯一的入口点。JASS VM 调用 thunk → thunk push native_id → thunk call dispatcher → dispatcher 查表 → 调用 Lua 回调。

分配器（ThunkAllocator, lua_japi.cpp:51）：VirtualAlloc(PAGE_EXECUTE_READWRITE) 分配 4KB RWX 页，thunk 紧密排列（每个 14 字节），页满后分配新页。通过 call_rel 字段在分配后修正相对偏移。

8. nfunction_add 定位

native_registry.cpp:78-84 — 使用 "StringCase" native 注册块定位 nfunction_add：

  1. search_string("StringCase") → 在 .text 中找到引用 "StringCase" 的指令地址
  2. ptr += 9 → 跳过 push(5) + mov edx(5) = 10 字节…… 实际是跳过到 CALL 指令处
  3. convert_function(ptr) → 解析 E8 相对调用，得到 nfunction_add 的绝对地址
为什么选 "StringCase"？ 这是一个普通的 JASS native，其注册块中的 call nfunction_add 指令指向所有 native 共用的同一个注册函数。任何一个 native 都可以用来定位它。
